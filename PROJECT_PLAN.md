## 1. 프로젝트 개요

### 1.1 배경 및 목적
현대오토에버(VCP 프로젝트)에서 퍼블리싱 코드 전달 시 매번 수작업으로 검수하던 과정을 자동화합니다.
기존에 오토에버로부터 받은 피드백 메일 기준과 일반적인 퍼블리싱 코드 정리 기준을 기반으로,
코드의 문제점을 자동으로 찾아내고 수정 방법을 문서화합니다.

### 1.2 핵심 가치
- **비개발자 친화적**: 기획자도 이해할 수 있는 쉬운 언어로 결과 보고서 제공
- **확장 가능**: 검수 기준을 나중에 쉽게 추가/수정 가능
- **자동화**: ZIP 파일 업로드 → 검수 → 리포트 생성까지 자동

### 2.2 검수 항목 (오토에버 피드백 기반)

#### [기준 1] 폰트/타이포그래피 관련 정리
> **무엇이 문제인가?**
> 코드 곳곳에 같은 글자 크기, 굵기, 줄 간격 값이 반복해서 직접 입력되어 있을 때

- **규칙 1-1**: `fontSize`, `fontWeight`, `lineHeight` 같은 타이포그래피 속성이 하나의 컴포넌트 내에 **10개 이상** 직접 입력된 경우 → 테마 변수로 교체 권고
- **규칙 1-2**: 동일한 타이포그래피 조합이 **5개 이상의 컴포넌트**에서 중복 사용될 경우 → 공통 컴포넌트로 분리 권고
- **규칙 1-3**: 컴포넌트화, 모듈 경로 변경 등 수정 사항 정리

**Before/After 예시:**
```jsx
// ❌ Before: 직접 하드코딩된 값
<Box sx={{ fontSize: '14px', fontWeight: 600, lineHeight: '20px' }}>

// ✅ After: 디자인 시스템 토큰 사용
<Box sx={{ ...typography.body2 }}>
```

---

#### [기준 2] 색상 관련
> **무엇이 문제인가?**
> 코드에 색상 코드(#111111 같은 형태)가 직접 입력되어 있을 때

- **규칙 2-1**: `#111111`, `#E9EAEC`, `#8333E6` 등 특정 색상 코드가 코드베이스 전체에서 **50개 이상** 중복 사용된 경우 → HMG Design System CSS 변수로 교체 권고
- **규칙 2-2**: 디자인 시스템에 정의된 색상인 경우 반드시 변수(`colors.primary` 등) 사용

**Before/After 예시:**
```jsx
// ❌ Before: 하드코딩된 색상값
background: 'linear-gradient(180deg, #8333E6 40%, rgba(131, 51, 230, 0.85) 100%)'

// ✅ After: 테마에서 가져온 그라디언트
import { gradients } from '@/shared/theme';
background: gradients.purpleVertical
```

---

#### [기준 3] !important 사용
> **무엇이 문제인가?**
> CSS에서 `!important`는 다른 스타일을 강제로 덮어쓰는 명령인데, 이것이 많으면 코드 유지보수가 어려워집니다

- **규칙 3-1**: `!important` 사용 횟수 집계 → 과다 사용 시 경고
- **규칙 3-2**: `!important` 없이 해결할 수 있는 방법 제안

---

#### [기준 4] 스크롤바 스타일링
> **무엇이 문제인가?**
> 웹킷(Chrome, Safari) 전용 스크롤바 스타일만 있고 다른 브라우저 대응이 없는 경우

- **규칙 4-1**: `::-webkit-scrollbar` 사용 시 표준 `scrollbar-width` 속성 함께 사용 여부 확인
- 권장 방식: `scrollbarSx` 유틸리티를 `@/shared/theme`에서 가져와 재사용

---

#### [기준 5] 일반 코드 품질 (추가 예정)
> 추후 새로운 피드백이 올 때마다 여기에 기준을 추가합니다

## 3. 시스템 아키텍처

### 3.1 전체 흐름
```
사용자 (기획자)
    ↓  ZIP 파일 선택 (퍼블리싱 코드)
┌─────────────────────────────┐
│    웹 UI (브라우저에서 실행)    │
│  - 파일 업로드                │
│  - 검수 진행 상황 표시          │
│  - 결과 리포트 보기            │
└────────────┬────────────────┘
             ↓
┌─────────────────────────────┐
│    검수 엔진 (Python)         │
│  - ZIP 압축 해제              │
│  - 파일 스캔 (HTML/CSS/JS/TSX)│
│  - 규칙별 검수                │
│  - 결과 취합                  │
└────────────┬────────────────┘
             ↓
┌─────────────────────────────┐
│    검수 리포트 (HTML)          │
│  - 전체 요약                  │
│  - 항목별 문제점               │
│  - Before/After 코드          │
│  - 수정 이유 설명 (쉬운 언어)   │
└─────────────────────────────┘
```

### 3.2 기술 스택
| 구분 | 기술 | 이유 |
|------|------|------|
| 백엔드 | Python 3.11+ | 빠른 개발, 코드 파싱에 적합 |
| 웹 UI | Flask + HTML/CSS/JS | 가볍고 별도 설치 없이 브라우저 실행 |
| 코드 파싱 | Python 내장 (`re`, `ast`) | 정규식으로 패턴 매칭 |
| 리포트 | Jinja2 템플릿 | HTML 리포트 자동 생성 |
| 패키징 | PyInstaller (추후) | 설치 없이 EXE 실행 가능하게 |

## 7. 주요 고려사항

### 7.1 비개발자를 위한 설계 원칙
1. **전문 용어 최소화**: "컴포넌트", "import" 같은 용어는 괄호 안에 쉬운 설명 추가
2. **시각적 명확성**: 🔴 빨간색 = 심각한 문제, 🟡 노란색 = 주의, 🟢 초록색 = 양호
3. **왜 문제인지** 먼저 설명, 그 다음 수정 방법 제시
4. **클릭 최소화**: 파일 올리고 버튼 하나만 누르면 완료

### 7.2 확장 가능 구조
- 새로운 검수 기준을 추가할 때 `src/rules/` 폴더에 파일 하나만 추가하면 됨
- 각 규칙은 독립적으로 켜고 끌 수 있음
- 임계값(예: 50개 이상의 중복 색상)은 설정 파일로 조정 가능

### 7.3 검수 대상 파일 형식
- `.tsx`, `.ts` - React + TypeScript 컴포넌트
- `.jsx`, `.js` - React JavaScript 컴포넌트
- `.css`, `.scss` - 스타일시트
- `.html` - HTML 파일
